# 电商智能助手技术选型方案

**类 Shopify Sidekick · 多模型无关 · 双 Agent 方案对比**

基于 AWS + Aurora MySQL + K8s + GraphQL/REST | 参考 Shopify Engineering 生产实践（ICML 2025）

> v4.0 | 2026年4月18日 | 内部使用

---

## 一、项目背景与约束条件

本文档为电商 SaaS 平台的智能助手产品（类 Shopify Sidekick）提供完整的技术选型方案。架构设计参考了 Shopify Engineering 团队在 ICML 2025 发表的《Building Production-Ready Agentic Systems》。文档保留两种 Agent 架构方案供团队评估选择。

### 1.1 现有基础设施约束

| 约束项 | 详细信息 |
|--------|----------|
| 云平台 | AWS（已有基础设施） |
| 业务数据库 | Aurora MySQL（Sidekick 不直接访问，通过平台 API 读写） |
| 平台 API | REST API + GraphQL 双协议（读用 GraphQL，写用 REST） |
| 部署方式 | Kubernetes (K8s) |
| 产品形态 | 独立产品，通过 JS SDK 嵌入现有商家后台 |
| 模型策略 | 模型无关，支持 Claude / GPT / 通义千问 / DeepSeek |
| 团队规模 | 10 人以上 |
| 目标用户 | 1 万+ 商家 |

### 1.2 核心设计原则

- **不碰业务数据库**：Sidekick 仅通过现有平台 API 读写数据，不直接访问 Aurora MySQL
- **读用 GraphQL、写用 REST**：数据查询走 GraphQL 精准取数，写操作走 REST 语义明确
- **模型无关**：所有 Agent 代码不绑定任何模型供应商，通过 LiteLLM 路由层统一调度
- **最小引入**：不过度设计，每一个组件都有明确的引入理由

## 二、公共技术栈（两种方案共用）

无论选择哪种 Agent 架构，以下技术栈均为共用层，不受 Agent 架构选择影响。

| 层次 | 职责 | 技术选型 | 核心理由 |
|------|------|----------|----------|
| 前端 | 商家交互界面 | Next.js + Vercel AI SDK + 嵌入 JS SDK | 流式渲染 + 一行代码接入现有后台 |
| API 服务 | 认证、限流、调度 | FastAPI + Celery + LiteLLM | 异步支持佳 + 模型统一路由 |
| 模型 API | 多模型路由与分级 | LiteLLM → Claude/GPT/千问/DeepSeek | 多模型灵活切换 + 故障转移 |
| 数据检索 | 知识库、缓存、记忆 | Qdrant + Aurora PG(小) + ElastiCache Redis | MySQL 无向量能力，Qdrant 独立部署 |
| MCP 工具 | 平台 API 封装 | 1 个 GraphQL 读工具 + 5-6 个 REST 写工具 | 读用 GraphQL 精准取数，写用 REST |
| 可观测性 | 追踪、评估、监控 | LangSmith/Langfuse + RAGAS + DeepEval + CloudWatch | Agent 追踪 + RAG 评估 + AWS 监控统一 |
| 基础设施 | 计算、存储、CI/CD | K8s + Helm + ArgoCD + Terraform | 与现有 K8s 集群统一管理 |

### 前端接入层

| 组件 | 技术 | 说明 |
|------|------|------|
| 主框架 | Next.js 15 (App Router) | SSR + 流式渲染 |
| AI 对话组件 | Vercel AI SDK | 流式响应、工具调用 UI、多模型支持 |
| 嵌入 SDK | Vanilla JS Widget | 商家后台加一行 `<script>` 即可接入 |
| 图表 | Recharts / ECharts | Agent 生成的分析报告中嵌入交互式图表 |

### API 服务层

Sidekick 不直接访问商家的 MySQL 数据库，而是通过现有平台的 API 读写数据。前端与 API 服务之间使用两种协议：流式对话走 SSE（Server-Sent Events），普通操作走 HTTP REST。

| 组件 | 技术 | 说明 |
|------|------|------|
| API 服务 | FastAPI (Python) | 异步支持佳，与 LangGraph 同语言生态 |
| 异步队列 | Celery + Redis | 批量内容生成、定时分析等长任务 |
| 模型路由 | LiteLLM | 统一所有模型 API 格式 + 故障转移 + 成本统计 |
| 平台 API 网关 | HTTP Client + 商家 token | 调用现有平台 API 时使用商家授权的 token |

### 模型 API 层：多模型无关架构

三层抽象实现代码与模型完全解耦：① LiteLLM 统一所有模型 API 格式；② PydanticAI 定义 Agent 时不指定模型，运行时注入；③ Prompt 模板引擎按模型加载不同版本的提示词。

**模型提供商池：**

| 提供商 | 主要模型 | 接入方式 | 优势场景 |
|--------|----------|----------|----------|
| Anthropic | Claude Opus/Sonnet/Haiku | Bedrock 或直连 API | 推理、工具调用、安全 |
| OpenAI | GPT-5.4 / GPT-4o / 4o-mini | 直连 API 或 Azure | 生态最广、多语言写作 |
| 阿里百炼 | Qwen-Max / Qwen-Turbo | 百炼 API | 中文最佳、国内合规、成本低 |
| DeepSeek | DeepSeek-V3 / R1 | 直连 API | 性价比极高 |

**路由策略（YAML 可配置，不硬编码）：**

| 任务类型 | 首选模型 | 备用模型 | 选择理由 |
|----------|----------|----------|----------|
| 意图识别 | Qwen-Turbo | Haiku / GPT-4o-mini | 调用量最大，用最便宜的 |
| 数据分析 | Claude Sonnet | GPT-4o / Qwen-Max | 工具调用 + 推理能力强 |
| 内容生成 | GPT-4o | Claude Sonnet / Qwen-Max | 多语言写作质量高 |
| 中文商家 | Qwen-Max | Claude Sonnet | 中文理解最佳 + 国内合规 |
| 策略决策 | Claude Opus | GPT-5.4 | 最强推理，低频使用 |

支持按商家地区路由：国内商家默认用千问（合规 + 中文优化），海外商家用 Claude/GPT。

### 数据与检索层

| 组件 | 技术 | 存储内容与理由 |
|------|------|----------------|
| 业务数据访问 | 平台 GraphQL + REST API | 读操作用 GraphQL 精准取数，写操作用 REST。不直接查 MySQL |
| 向量搜索 | Qdrant（自托管 on EC2） | 商品知识库、帮助文档的向量索引。MySQL 无向量能力 |
| Sidekick 内部数据 | Aurora PostgreSQL（小实例） | Agent 状态检查点、会话历史、评估数据、Mem0 偏好 |
| 缓存与会话 | ElastiCache Redis | 语义缓存（降本 40-68%）+ 会话状态 + 商家级限流 |
| 长期记忆 | Mem0 + Aurora PG | 跨会话商家偏好记忆 |
| 检索策略 | Hybrid RAG + Adaptive RAG | 简单问题走 GraphQL API，复杂问题走 RAG 知识库检索 |

## 三、核心决策：单 Agent vs 多 Agent 方案对比

这是整个技术选型中最关键的架构决策。两种方案共用上述所有技术栈，区别仅在 Agent 编排层的设计。

### 3.1 架构对比

| 维度 | 方案 A：单 Agent + JIT 指令 | 方案 B：多 Agent 编排 |
|------|---------------------------|---------------------|
| 核心思路 | 一个 Agent 带所有工具，工具返回数据时附带场景指令动态切换行为 | Manager Agent 识别意图后路由到专业 Agent，每个 Agent 有独立指令和工具 |
| 编排框架 | LangGraph 做 Agentic Loop（循环调用工具直到完成）+ Human-in-the-loop | LangGraph 做有向图编排（节点=Agent，边=流转条件） |
| 单 Agent 定义 | PydanticAI 定义一个 Agent，挂所有工具 | PydanticAI 定义 4 个专业 Agent（Manager/Analytics/Content/Operations） |
| 场景区分 | 通过 JIT 指令：工具返回销售数据时附带"分析师指令"，返回商品数据时附带"内容创作指令" | 通过 Agent 分工：分析任务交给 Analytics Agent，内容任务交给 Content Agent |
| LLM 调用次数 | 较少：无 Manager 路由调用，直接开始工作 | 较多：Manager 先调一次 LLM 识别意图，再调专业 Agent |
| 系统提示词 | 精简：只放通用指令，场景指令由工具动态注入（JIT） | 每个 Agent 有完整的专业指令，但可能膨胀 |
| Shopify 实践 | ✅ Shopify Sidekick 采用此方案，经过生产验证。注意：Shopify 能用单 Agent 管 50+ 工具，靠的是 JIT 指令 + GRPO 模型微调两个条件（详见 3.6） | ⚠️ Shopify 明确建议"早期避免多 Agent"，但其前提是经过 GRPO 微调的模型对工具使用准确率极高（~99%） |

### 3.2 优劣势对比

| 维度 | 方案 A：单 Agent + JIT | 方案 B：多 Agent |
|------|----------------------|----------------|
| 开发速度 | ✅ 快——只需定义一个 Agent + JIT 指令，无需设计 Agent 间通信 | ⚠️ 慢——需定义 4 个 Agent、路由逻辑、状态传递 |
| 调试难度 | ✅ 简单——只跟踪一个 Agent 的决策链路 | ⚠️ 复杂——要跟踪多个 Agent 之间的交互和状态流转 |
| 成本 | ✅ 低——少一次 Manager 的 LLM 调用，每次交互省一次意图识别 | ⚠️ 较高——Manager 每次都要调 LLM 识别意图 |
| 工具扩展性 | ⚠️ 工具超过 20 个后可能出现混乱。Shopify 能管 50+ 工具靠的是 GRPO 微调，我们不做微调，上限更低 | ✅ 每个 Agent 只看到自己的工具，可无限扩展 |
| 指令冲突 | ⚠️ 可能冲突——分析指令和内容创作指令可能互相干扰 | ✅ 不冲突——每个 Agent 有独立的指令空间 |
| 崩溃恢复 | ✅ 简单——只有一个 Agent 的状态需要保存 | ✅ 强大——LangGraph 每个节点都有检查点，更精确 |
| 团队分工 | ⚠️ 难并行——所有人改同一个 Agent | ✅ 易并行——4 个人/组各负责一个 Agent |

### 3.3 同一场景执行流程对比

**场景：商家说"上周哪些商品销量下滑了，帮我给这些商品写新的促销文案"**

#### 方案 A：单 Agent + JIT

| # | 执行步骤 |
|---|----------|
| 1 | Agent 理解意图，决定先查销售数据 → 调用 query_store_data（查 sales） |
| 2 | 工具返回：销售数据 + JIT 指令"对比周环比变化，标注涨跌超 10% 的商品" |
| 3 | Agent 按 JIT 指令分析数据，输出下滑商品列表 |
| 4 | Agent 继续决定要写文案 → 调用 query_store_data（查 products 详情） |
| 5 | 工具返回：商品数据 + JIT 指令"SEO 描述控制在 160 字符内，参考同类商品风格" |
| 6 | Agent 生成促销文案，调用 save_content 保存 → 触发 Human-in-the-loop 确认 |
| 7 | 商家确认 → 完成。全程一个 Agent，无路由开销 |

**LLM 调用次数：3-4 次**（理解+分析+生成+确认）

#### 方案 B：多 Agent 编排

| # | 执行步骤 |
|---|----------|
| 1 | Manager Agent 调用 LLM 识别意图："分析 + 内容"，拆分为两个子任务 |
| 2 | LangGraph 路由到 Analytics Agent，传入商家上下文 |
| 3 | Analytics Agent 调用 query_store_data 查询数据，输出 SalesReport 结构化结果 |
| 4 | LangGraph 保存检查点，把分析结果传入 Content Agent |
| 5 | Content Agent 查询商品详情，生成促销文案（输出 PromotionContent 结构化结果） |
| 6 | LangGraph 路由到 Human-in-the-loop 节点，暂停等待商家确认 |
| 7 | 商家确认 → 调用 save_content 保存 → 完成 |

**LLM 调用次数：5-6 次**（Manager意图+分析+查询+生成+确认…），比方案 A 多 1-2 次

### 3.4 适用场景对比

| 场景 | 方案 A 表现 | 方案 B 表现 |
|------|------------|------------|
| POC / 早期产品 | ⭐ 最优——简单快速，2-3 周可出 Demo | 可行但较慢，4 周出 Demo |
| 工具数 < 15 个 | ⭐ 最优——完全可控，不会混乱 | 可行但过度设计 |
| 工具数 20-50 个 | 需警惕——JIT 可缓解但开始吃力 | ⭐ 最优——分 Agent 降低每个 Agent 的工具复杂度 |
| 场景指令频繁冲突 | 困难——JIT 指令可能互相干扰 | ⭐ 最优——Agent 隔离彻底解决冲突 |
| 10+ 人团队并行开发 | 困难——所有人改同一个 Agent | ⭐ 最优——每个组负责一个 Agent |
| 对延迟和成本敏感 | ⭐ 最优——少 1-2 次 LLM 调用 | Manager 路由增加额外调用和延迟 |

### 3.5 推荐策略：分阶段演进

两种方案不是二选一，而是分阶段递进：

| 阶段 | 架构 | 触发条件 | 工作量 |
|------|------|----------|--------|
| POC | **方案 A：单 Agent + JIT** | 立即开始 | 2-3 周出 Demo |
| 上线 v1 | **方案 A：单 Agent + JIT** | 工具 < 15 个且无指令冲突 | 优化 JIT 指令和评估 |
| 规模化 | **方案 B：拆分多 Agent** | 工具超 20 个 / 指令冲突频发 / 团队需要并行 | 从单 Agent 拆出专业 Agent |

从方案 A 迁移到方案 B 的成本很低：PydanticAI 定义的 Agent 可以直接拆分，LangGraph 的 Agentic Loop 可以平滑升级为多节点图编排，MCP 工具完全复用。架构上的"插槽"已预留。

### 3.6 关键差异：Shopify 做了模型微调，我们没有

Shopify 的单 Agent 之所以能管 50+ 工具，依赖两个条件：

1. **JIT 指令**：工具返回数据时附带场景指令，动态切换 Agent 行为。这个我们可以直接学习并应用。
2. **GRPO 模型微调**：Shopify 使用 Group Relative Policy Optimization（强化学习）对模型进行专项训练，让模型精确理解每个工具的使用场景。微调后语法验证准确率从 ~93% 提升到 ~99%，LLM Judge 相关性从 0.66 提升到 0.75。

我们的方案是调用第三方模型 API（Claude/GPT/千问），不做模型微调。这意味着：

| 维度 | Shopify（有微调） | 我们（无微调） |
|------|-------------------|---------------|
| 模型对工具的理解 | 专门训练过，准确率 ~99% | 依赖通用模型能力，准确率较低 |
| 单 Agent 工具上限 | 50+ 工具仍可控 | 保守估计 15-20 个为安全上限 |
| 错误率控制 | GRPO 训练直接降低错误 | 靠 PydanticAI 结构化验证 + 重试机制补偿 |
| 投入成本 | 需要 ML 团队 + GPU 训练资源 | 零额外成本 |

**对我们的实际影响**：当前工具总数不到 10 个（1 个 GraphQL + 5-6 个 REST），远低于任何危险线，通用模型完全够用。但如果未来工具扩展到 20+ 个，需要认真评估是投入做微调（走 Shopify 路线）还是拆分为多 Agent（走方案 B 路线）。对于不做微调的团队，方案 B 的拆分门槛应该比 Shopify 建议的更早触发。

## 四、JIT 指令模式（来自 Shopify 实践）

JIT（Just-in-Time）指令是 Shopify Sidekick 的核心架构突破。核心思想：不把所有指导塞进系统提示词，而是在工具返回数据时附带场景相关的指令。目标是为 LLM 构建每个场景的完美上下文——不多一个 token，不少一个 token。

### 4.1 实现方式

JIT 指令嵌入在 MCP 工具的返回结果中。工具根据查询内容自动附带对应的指令：

| 工具查询内容 | 附带的 JIT 指令 | 效果 |
|-------------|----------------|------|
| 查询销售数据 | "对比周环比，标注涨跌超 10% 的商品，分析可能原因" | Agent 自动切换为"分析师模式" |
| 查询库存数据 | "库存低于 7 天销量的商品标记预警，建议补货" | Agent 自动切换为"运营顾问模式" |
| 查询商品详情 | "SEO 描述控制 160 字符，参考同类商品风格" | Agent 自动切换为"内容创作者模式" |
| 调用写操作工具 | "此操作不可撤销，必须向商家展示确认卡片" | Agent 自动触发 Human-in-the-loop |

### 4.2 三个核心优势

- **局部化指导**：指令只在相关时出现，系统提示词只保留基础 Agent 行为定义
- **缓存友好**：动态调整指令不会破坏 LLM 的 prompt cache，因为系统提示词不变
- **模块化**：不同指令可根据 beta flag、模型版本、页面上下文动态调整

注意：JIT 指令在方案 A（单 Agent）中是核心机制，在方案 B（多 Agent）中同样有价值——可以用来细化每个 Agent 内部的场景行为。

## 五、GraphQL + REST 双协议策略

### 5.1 为什么读操作用 GraphQL

- 一次查询精准获取所需数据，避免多次 REST 调用。商家问"上周哪些商品销量下滑且库存积压"，GraphQL 一次查询跨两个实体拿到结果，REST 需要 3-5 次调用
- 只返回需要的字段，减少 70-80% 冗余数据，直接降低模型 token 成本
- 强类型 Schema 天然就是 Agent 的工具定义，通过 Introspection 自动发现可用数据
- LLM 生成 GraphQL 查询的能力很强，因为 GraphQL 本身就是结构化、强类型的

### 5.2 为什么写操作用 REST

- POST/PUT/DELETE 语义明确，审计日志更清晰
- 幂等性和重试逻辑更成熟，适合需要 Human-in-the-loop 确认的场景
- 现有平台的 REST 权限控制通常比 GraphQL Mutation 更细粒度

### 5.3 MCP 工具设计

数据查询只需一个通用 GraphQL 工具，写操作每个动作一个 REST 工具。相比纯 REST 方案的几十个工具，工具总数缩减至 6-8 个，Agent 选对工具的准确率大幅提升。

| 工具名 | 协议 | 权限 | 用途 | 说明 |
|--------|------|------|------|------|
| query_store_data | GraphQL | 只读 | 所有数据查询 | Agent 生成 GraphQL 查询语句 |
| create_promotion | REST POST | 写（需确认） | 创建促销活动 | Human-in-the-loop |
| update_price | REST PUT | 写（需确认） | 修改商品价格 | Human-in-the-loop |
| update_inventory | REST PUT | 写（需确认） | 更新库存 | Human-in-the-loop |
| save_content | REST PUT | 写（需确认） | 保存 AI 生成的商品描述 | Human-in-the-loop |
| create_automation | REST POST | 写（需确认） | 创建自动化规则 | Human-in-the-loop |

GraphQL Schema 可通过 Introspection 自动注入 Agent 的系统提示词。当平台新增字段或实体时，Agent 自动可用，无需改代码。

## 六、评估体系（来自 Shopify 实践）

Shopify 强调："vibe testing is not enough"——必须有统计严谨性的评估体系。

### 6.1 四层评估架构

| 评估层 | 工具 | 衡量什么 | 运行时机 |
|--------|------|----------|----------|
| 检索质量 | RAGAS | 检索精度、忠实度、答案相关性 | 每次部署前 CI/CD 自动跑 |
| 安全护栏 | DeepEval | 幻觉、毒性、偏见、Reward Hacking | 每次部署前 + 生产抽检 |
| LLM-as-Judge | LangSmith / Langfuse | 回答质量、工具使用正确性、用户满意度 | 生产环境持续监控 |
| 用户模拟 | LLM 模拟商家 | 端到端场景完整性、边界情况覆盖 | 重大变更前自动回归测试 |
| 人工抽检 | 产品专家团队 | 写操作正确性、商业合理性 | 定期抽样审核 |

### 6.2 Reward Hacking 防范（Shopify 的教训）

Shopify 发现模型会以创造性方式绕过评估系统：

- **回避黑客**：遇到难题时选择"解释为什么做不了"而不是尝试做
- **标签黑客**：用客户标签作为万能匹配，而不是使用正确的字段映射
- **Schema 违规**：幻觉生成 ID 或使用错误的枚举值

应对措施：在 DeepEval 的评估指标中加入"回避检测"——检查模型是否在可以做的任务上选择了回避。同时用 PydanticAI 的结构化输出验证来捕捉 Schema 违规。Shopify 通过迭代改进语法校验器和 LLM Judge 来应对这些问题，最终语法验证准确率从 ~93% 提升到 ~99%。我们不做 GRPO 微调，但可以通过 PydanticAI 的类型验证 + 自动重试机制达到类似的错误捕捉效果。

### 6.3 用户模拟测试

在 CI/CD 中加入 LLM 模拟商家行为的测试层：用 LLM 扮演不同类型的商家（新手、高级用户、故意测试边界的），自动跑一轮对话，用 RAGAS 和 DeepEval 评估结果。这是 Shopify 在上线前捕捉回归和验证改进的核心手段。

## 七、K8s 部署架构

Sidekick 在现有 K8s 集群中作为独立 namespace（sidekick）运行：

| Deployment | 副本数 | 资源 | HPA | 说明 |
|------------|--------|------|-----|------|
| sidekick-api | 3-10 | 1C / 2Gi | CPU>70% | FastAPI 主服务 |
| sidekick-worker | 2-8 | 2C / 4Gi | KEDA 队列深度 | Celery 异步任务 |
| sidekick-mcp | 2-6 | 0.5C / 1Gi | CPU>60% | MCP Server |
| qdrant | 1-3 | 2C / 8Gi | 手动 | StatefulSet + PVC |

所有服务通过 Helm Charts 管理，配置通过 ConfigMap 和 Secrets 注入。CI/CD 用 GitHub Actions 构建 + RAGAS 评估门禁，ArgoCD GitOps 部署。

K8s Ingress 需注意 SSE 超时配置：Agent 复杂分析可能超过默认 60 秒超时，需调大 Ingress 的响应超时。

## 八、成本估算（1 万商家）

假设日活 30%（3000 商家/天），均日 5 次对话。两种方案的成本差异主要在模型 API 调用上：

| 项目 | 方案 A（单 Agent） | 方案 B（多 Agent） | 差异原因 |
|------|-------------------|-------------------|----------|
| 模型 API | $4,000-6,000 | $5,000-8,000 | 方案 B 多 Manager 路由调用 |
| K8s 计算 | $1,500-2,500 | $1,500-2,500 | 基本一致 |
| 数据层（PG + Redis + Qdrant） | $400-800 | $400-800 | 基本一致 |
| 可观测性 | $0-500 | $0-500 | 基本一致（Langfuse 自托管可降至零） |
| **月总计（优化后）** | **$6,000-10,000** | **$7,500-12,000** | **方案 A 月省 $1,500-2,000** |

成本优化关键手段：模型分级调用（意图识别用最便宜的模型）、Redis 语义缓存（降本 40-68%）、GraphQL 精准取数（减少 70-80% 冗余 token）。

## 九、POC 与生产环境差异

POC 阶段用最简配置快速验证，上线前补齐以下 8 项：

| # | 事项 | POC 状态 | 生产要求 |
|---|------|----------|----------|
| 1 | 多租户数据隔离 | 单租户测试 | 商家级 token 隔离 + 审计 |
| 2 | Agent 状态持久化 | 内存状态 | PG Checkpointer + 崩溃恢复 |
| 3 | 语义缓存 | 无缓存 | ElastiCache Redis 语义缓存 |
| 4 | 模型路由 + 故障转移 | 单模型直连 | LiteLLM 多模型路由 + 自动切换 |
| 5 | 模型分级调用 | 全用一个模型 | 按任务复杂度分级调用 |
| 6 | 操作审计日志 | 无审计 | 每次写操作记录完整审计 |
| 7 | K8s 自动扩缩容 | 固定副本 | HPA + KEDA 自动扩缩容 |
| 8 | 可观测性完善 | LangSmith 免费层 | 付费版/自托管 + RAGAS + 成本报表 |

其中第 1-3 项建议在 POC 阶段就做好骨架，第 4-8 项在上线前集中补齐。

## 十、本地开发环境

全套技术栈可本地搭建。`docker-compose.yml` 拉起 PostgreSQL + Redis + Qdrant，其余组件通过 `pip install` 安装。代码与生产环境完全一致，只有配置文件中的连接地址和 API key 不同。现有平台 API 用 FastAPI Mock Server 模拟。离线开发可用 Ollama 本地小模型替代云端 API。

## 十一、一句话总结

**商家在现有后台通过 JS SDK 弹出 Sidekick 对话框 → FastAPI 认证后转给 LangGraph Agentic Loop → 单个 Agent（PydanticAI 定义）带全部工具，通过 JIT 指令动态切换场景行为 → 读操作用 GraphQL 精准取数，写操作用 REST（需商家确认）→ LiteLLM 按任务复杂度路由到 Claude/GPT/千问等不同模型，按商家地区智能切换 → Redis 语义缓存降低 40-68% 调用成本，Mem0 记住商家偏好 → 全程被 LangSmith 追踪 + RAGAS 评估质量 → 部署在现有 K8s 集群的独立 namespace 中。当工具超过 20 个或场景指令冲突时，从单 Agent 平滑拆分为多 Agent 编排（注：Shopify 单 Agent 管 50+ 工具靠的是 JIT + GRPO 模型微调，我们不做微调，拆分门槛应更早）。**
